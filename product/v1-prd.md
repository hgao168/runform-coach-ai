## 一、产品愿景

> 跑者打开 RunForm → 点击「开始跑步」→ 手机放在腰包/臂带中 → 跑步过程中通过耳机听到实时语音提示：「步频 168，偏低，试着加快到 175」「注意挺直躯干，你有点前倾了」→ 跑步结束后收到完整的姿态总结报告，看到自己的进步趋势。

RunForm v1 的核心使命：**从「上传视频做分析」进化到「打开 App 跑起来就有实时语音教练」**。这是品类定义级别的跃迁——五个主流竞品（Runna、NRC、Strava、Keep、咕咚）中没有一个能做到「跑步中实时感知并矫正跑姿」。

**差异化定位**（来自竞品扫描）：

```
┌─────────────────────────────────────────────────────────┐
│               RunForm 蓝海定位                          │
│                                                         │
│  "AI that understands HOW your body runs"               │
│                                                         │
│  从「你跑了多少」→「你的身体怎么跑」                      │
│  从「通用训练计划」→「基于你实际姿态的个性化计划」         │
│  从「事后分析」→「跑步中实时矫正」                        │
│                                                         │
│  五个竞品中没有任何一个做到这三点。                       │
└─────────────────────────────────────────────────────────┘
```

## 二、用户画像（Personas）

### P1：认真跑者 小陈（核心目标用户）
- **年龄**：28 岁
- **跑步水平**：半马 PB 1:45，月跑量 120km
- **设备**：iPhone 14 + AirPods
- **痛点**：跑完膝盖隐隐作痛，不确定是跑姿问题还是训练量问题；网上搜「正确跑姿」都是文字描述，不知道自己的跑姿到底对不对；想提高步频但不知道当下到底多少步频
- **需求**：跑步中实时知道自己的步频和姿态，得到可执行的纠正建议
- **使用场景**：每周 4 次跑步，每次 30-60 分钟，手机放腰包，戴耳机听语音提示

### P2：入门跑者 小李
- **年龄**：24 岁
- **跑步水平**：刚开始跑步 3 个月，5km 配速 7:00
- **设备**：iPhone 13
- **痛点**：跑步时完全凭感觉，不知道自己跑得对不对；看到网上说「步频要 180」但自己不知道在不在这个范围；担心跑姿不对导致受伤而放弃跑步
- **需求**：需要一个「陪跑教练」告诉他怎么改进，门槛低、易懂
- **使用场景**：每周 2-3 次跑步，每次 20-30 分钟，需要鼓励和基本指导

### P3：严肃跑者 老王
- **年龄**：42 岁
- **跑步水平**：全马 PB 3:30，月跑量 200km+
- **设备**：iPhone 15 Pro + Garmin 手表 + AirPods
- **痛点**：Garmin 和 Strava 能告诉他配速和心率，但不能告诉他跑姿；视频分析只有跑后看，不能在跑步中即时纠正；年龄增长后更关注伤病预防
- **需求**：跑步中实时姿态监测 + 跑后深度分析与 Kipchoge 对比
- **使用场景**：每周 5 次跑步，希望在不增加设备（已有手机）的前提下获取跑姿数据

## 三、功能层级（Must / Should / Could）

### 3.1 Must Have（v1 必须交付，否则不发版）

| # | 功能 | 用户价值 | 对应 Phase | 技术栈 |
|---|------|----------|-----------|--------|
| **M1** | **实时步频检测与语音播报** | 跑者最关心的第一指标（调查显示 80%+ 跑者最关心步频）；让用户在跑步中听到「当前步频 168 SPM」 | Phase 0 + 1 | CoreMotion 加速度计 → 低通滤波 + 峰值检测 → AsyncStream<CadenceUpdate> → AVSpeechSynthesizer 播报 |
| **M2** | **躯干姿态实时监测** | 跑步中不自觉前倾/后仰是最常见问题；实时感知并提示可有效预防腰背伤痛 | Phase 3 | 加速度计+陀螺仪互补滤波 → Pitch/Roll/Yaw → 躯干倾角 ± 语音触发 |
| **M3** | **垂直振幅 + 触地时间估算** | 步频之外第二重要的跑步经济性指标；触地时间过长意味着刹车效应增加 | Phase 2 | Z轴加速度双重积分（垂直振幅）+ 合成加速度范数谷值检测（触地时间） |
| **M4** | **跑步结束后生成姿态总结报告** | 完成「分析 → 反馈 → 改进」的完整闭环；复用现有 AnalysisResultView 展示 | Phase 5 | RunSession → PoseMetrics 兼容格式 → AnalysisResultView |
| **M5** | **跑步历史记录与趋势对比** | 看到进步是用户留存的核心驱动力 | Phase 5 | 复用现有 HistoryView + HistoryTrendComponents，新增「实时跑步」会话类型 |

### 3.2 Should Have（v1 尽力交付，可降级到 v1.1）

| # | 功能 | 用户价值 | 降级方案 |
|---|------|----------|----------|
| **S1** | **步幅/跨步风险评估** | 过度跨步是跑步伤病第一大诱因 | 若传感器精度不足，标记为「实验性功能」或移到 v1.1 |
| **S2** | **着地方式提示（前掌/全掌/后跟）** | 跑者热门话题，从加速度冲击特征可做初步推断 | 降级为仅在跑后报告中显示 |
| **S3** | **跑步中姿态评分实时更新** | 游戏化体验，激励持续改善（如「当前跑姿评分 78/100」） | 跑后报告中显示，跑步中不实时更新 |
| **S4** | **自定义目标步频/配速区间** | 满足不同水平跑者需求（入门 160-170，进阶 170-180） | v1 使用默认值 170-180 SPM |

### 3.3 Could Have（v2 考虑，v1 不做）

| # | 功能 | 推迟理由 |
|---|------|----------|
| **C1** | Apple Watch 独立 App | Watch 开发增加 2-4 周工作量，且限制用户群。先用 iPhone 验证 PMF |
| **C2** | 左右脚对称性实时分析 | 需要更高精度的 IMU 数据 + 算法研发周期长 |
| **C3** | 跑步视频 + IMU 数据融合分析 | 前期先分别跑通两个独立管线，融合放在 v2 |
| **C4** | 伤病风险实时预警 | 需要积累足够训练数据 + 临床验证，不能做不成熟的预警功能 |
| **C5** | 实时语音教练对话（用户用语音提问） | Siri/语音识别集成 + NLP 理解，技术复杂度高，v1 核心体验验证后再做 |

## 四、验收标准（Acceptance Criteria）

### M1：实时步频检测与语音播报

| # | 验收标准 | 验证方式 |
|---|----------|----------|
| M1-AC1 | 启动跑步会话后，步频值在 3 秒内开始显示（延迟 ≤ 3s） | 自动测试：模拟 100Hz 加速度计数据流，验证首帧 CadenceUpdate 时间戳 ≤ 3s |
| M1-AC2 | 手动击掌 180bpm 验证步频误差 ≤ ±2 SPM | 手动测试：iPhone 绑在腰间，以 180bpm 节奏击掌，验证显示值 178-182 SPM |
| M1-AC3 | 步频语音播报间隔 ≥ 15 秒（防抖） | 日志验证：相邻两条步频提示的时间戳间隔 ≥ 15s |
| M1-AC4 | 每次跑步步频语音提示 ≤ 5 条 | 日志验证：RunSession 内 AudioCoachManager 记录不超过 5 条步频提示 |
| M1-AC5 | 中文 TTS 语音清晰可辨识（使用 Ting-Ting 语音包） | 人工验收：3 人试听，清晰度评分 ≥ 4/5 |
| M1-AC6 | 功耗：100Hz 持续采集 5 分钟，电池消耗 < 3% | 仪器测试：iPhone 14 从 100% 开始，跑步 5 分钟后电量 ≥ 97% |

### M2：躯干姿态实时监测

| # | 验收标准 | 验证方式 |
|---|----------|----------|
| M2-AC1 | 躯干前倾角度在正常站姿下显示 0° ± 3° | 手动测试：手机竖直绑在腰部，站直，验证 Pitch ≈ 0° |
| M2-AC2 | 跑步机上跑步时，躯干前倾角度在 5-15° 范围（符合生物力学文献） | 跑步机测试：慢跑 3 分钟，采集 Pitch 均值，验证在 5-15° 范围 |
| M2-AC3 | 连续跑步 10 分钟，无姿态数据丢失（丢帧率 < 2%） | 日志验证：SensorFrame 总量 vs 接收总量，丢帧率 < 2% |
| M2-AC4 | 姿态异常时语音提示触发（前倾 > 20° 或后仰 > 5°，持续 ≥ 3 秒） | 模拟测试：倾斜手机模拟异常姿态，验证 3 秒后触发语音提示 |

### M3：垂直振幅 + 触地时间估算

| # | 验收标准 | 验证方式 |
|---|----------|----------|
| M3-AC1 | 垂直振幅数值在合理范围（跑步时 3-13cm，文献参考值） | 跑步机测试：3 人慢跑，垂直振幅在 3-13cm |
| M3-AC2 | 触地时间数值在合理范围（慢跑 200-300ms，文献参考值） | 跑步机测试：3 人慢跑，GCT 在 200-300ms |
| M3-AC3 | 与视频 PoseExtractor 对比：步频误差 ≤ ±3 SPM | 同步录制跑步视频 + 传感器采集，PoseExtractor 结果 vs 传感器结果对比 |
| M3-AC4 | 30 分钟连续采集无内存泄漏 | Instruments Leaks 检测：30 分钟持续运行，无内存泄漏报告 |

### M4：跑步结束后姿态总结报告

| # | 验收标准 | 验证方式 |
|---|----------|----------|
| M4-AC1 | 跑步结束后自动跳转到总结页面，展示：总时长、平均步频、平均垂直振幅、平均触地时间、躯干倾角趋势图 | 端到端测试：完成一次 ≥ 5 分钟跑步 → 停止 → 总结页面正常渲染 |
| M4-AC2 | 总结页面显示「步频评级」（优秀 170-180 / 良好 160-170 / 需改进 < 160） | 数据驱动测试：输入不同平均步频，验证评级正确 |
| M4-AC3 | 总结页面可保存到历史记录 | 测试：停止跑步 → 点击保存 → 历史列表可见该记录 |

### M5：跑步历史记录与趋势对比

| # | 验收标准 | 验证方式 |
|---|----------|----------|
| M5-AC1 | 历史列表展示「实时跑步」类型会话，显示日期、时长、平均步频 | UI 测试：至少有 3 条历史记录时，列表正确渲染 |
| M5-AC2 | 趋势图展示步频变化（最近 10 次跑步） | 模拟数据测试：插入 10 条不同步频的历史记录，趋势图正确绘制 |
| M5-AC3 | 点击历史记录可查看完整跑步会话详情 | 端到端测试：历史列表点击 → 详情页显示所有指标 |

## 五、成功指标（Success Metrics）

### 5.1 北极星指标

**v1 发布后 30 天：周活跃跑者（至少完成 1 次实时跑步会话）≥ 500 人**

### 5.2 关键指标

| 指标 | 目标值 | 测量方式 | 监测频率 |
|------|--------|----------|----------|
| **D1 留存率** | Day 7 ≥ 40%，Day 30 ≥ 25% | Firebase / App Store Connect 分析 | 每周 |
| **D7 核心动作完成率** | 安装后 7 天内完成首次实时跑步的比例 ≥ 35% | 事件埋点：`run_session_started` → `run_session_completed` | 每周 |
| **步频改善率** | 连续使用 4 周后，平均步频提升 ≥ 5 SPM 的用户占比 ≥ 50% | 聚合查询历史跑步数据 | 每双周 |
| **NPS（净推荐值）** | ≥ 40（「好」级别） | 跑步结束后弹出 NPS 评分 | 每月 |
| **App Store 评分** | ≥ 4.5 星（前 30 天） | App Store Connect | 每周 |
| **Crash-free rate** | ≥ 99.5% | Firebase Crashlytics | 实时 |
| **语音提示满意度** | 「语音提示有帮助」的比例 ≥ 70% | 跑步后反馈弹窗 | 每周 |
| **传感器数据质量** | 丢帧率 < 2%，步频误差 < ±3 SPM | 服务端聚合日志 | 每周 |
| **功耗表现** | 30 分钟跑步平均电池消耗 < 15% | 客户端埋点 | 每版本 |

### 5.3 护栏指标（不应劣化）

| 指标 | 基线值 | 护栏 |
|------|--------|------|
| App 启动时间 | 当前 < 2s | v1 仍 < 2s |
| 视频分析功能 Bug 率 | 当前（无系统数据） | v1 不引入 ≥ P2 的视频分析回归 Bug |
| 后端 API 可用性 | ≥ 99.9% | v1 维持 |
| 后台挂起/被 kill 率 | N/A（新功能） | 30 分钟跑步会话被中断率 < 10% |

## 七、关键假设与验证计划

| # | 假设 | 验证方法 | 如果失败 |
|---|------|----------|----------|
| H1 | iPhone 放腰包中的步频检测精度能达到 ±2 SPM | Phase 1 验收测试（M1-AC2） | 提示用户使用臂带/紧身腰包；调研第三方 IMU 配件 |
| H2 | 手机姿态角可以可靠推断躯干前倾（误差 < 5°） | Phase 3 与视频 ground truth 对比 | 降级为仅步频+振幅；躯干角标记为「实验性」 |
| H3 | 跑者愿意在跑步中听到 ≤ 5 条语音提示 | Sprint 0 用户访谈 + Beta 测试反馈 | 调整为 3 条上限或仅节拍器模式 |
| H4 | iOS 后台跑步可持续 ≥ 30 分钟不掉数据 | Phase 5 压力测试 | 启用 HKWorkoutSession 确保后台活跃 |
