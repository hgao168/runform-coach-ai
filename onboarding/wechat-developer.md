### 2.3 云开发技术选型评估（AI 推理后端）

微信小程序的核心约束：
- ❌ **无后台录制**：小程序进入后台即暂停，无法录制跑步全程视频
- ❌ **无可编程相机帧访问**：`wx.createCameraContext()` 仅用于拍照/录像，无法逐帧分析
- ✅ **有加速度计**：`wx.onAccelerometerChange` 可获取实时加速度，适合步频检测
- ✅ **有录音权限**：`wx.getRecorderManager()` 可录制音频（呼吸节奏分析辅助）
- ❌ **无端侧 AI 推理**：`wx.createInferenceSession()` 仅支持有限模型格式，不支持 Pose Estimation
- ✅ **视频上传**：`wx.uploadFile` 支持 ≤10MB 视频上传，耗时约 30-60s（取决于网络）

结论：**AI 推理必须在云端进行。**

#### 方案对比

| 维度 | A: Railway (当前) | B: WeChat 云开发 (CloudBase) | C: Tencent Cloud 全能方案 |
|------|-------------------|------------------------------|--------------------------|
| **已投入成本** | ✅ 已有 /analyze、/training-plan、/athletes 接口 | ❌ 需迁移全部后端 | ❌ 需迁移全部后端 |
| **GPU AI 推理** | ✅ Railway 支持 GPU 实例（需升级套餐） | ❌ 云函数不支持 GPU；云托管可配置 CVM GPU 但复杂 | ✅ Tencent Cloud GPU CVM (GN10Xp 等) 原生支持 |
| **小程序集成** | 需配置 request 合法域名 | ✅ 免域名配置，天然集成 | 需配置域名 |
| **视频上传体验** | 上传到外部域名，需白名单 | ✅ 微信云存储 CDN 加速，上传更快 | ✅ COS SDK 直传，CDN 加速 |
| **用户登录** | 需自行实现 wx.login | ✅ 云函数 `cloud.getWXContext()` 天然获取 openid | 需自行实现 |
| **成本（月）** | Railway Pro ~$20 + GPU ~$0（按需） | 基础版免费，调用量超限后 ~¥200/月 | GPU CVM ~¥3,000-8,000/月起 |
| **运维复杂度** | 中等（需管理 Railway 部署） | 低（微信生态统一管理） | 高（需运维 GPU 实例） |
| **可迁移性** | ✅ 标准 FastAPI，可迁移到任何云 | ❌ 强绑定微信云开发 SDK | 中等（基于 CVM，可迁移） |

#### 推荐方案：渐进式混合架构

```
Phase 1 (Sprint 0-1)：保持 Railway 后端不变
  ├── 保留现有 /training-plan、/athletes 接口
  ├── 新增 /analyze-posture（Pose Estimation 视频分析）在 Railway
  │   └── 先用轻量方案：服务端 MediaPipe，CPU 推理（视频≤30s，处理时间 15-30s 可接受）
  └── 新增 /cadence-analyze 接口：接收加速度数据 → 步频/步幅估算

Phase 2 (Sprint 2-3)：引入 WeChat 云开发
  ├── 云函数：wx.login → openid 获取（用户体系）
  ├── 云数据库：用户分析历史（替代当前 wx.setStorageSync 本地存储）
  ├── 云存储：视频文件存储（替代每次上传到 Railway）
  └── 优势：免域名配置、自动 CDN、多端同步

Phase 3（GPU 升级）：如 CPU 推理延迟不可接受
  ├── 迁移 /analyze-posture 到 Tencent Cloud GPU CVM
  └── 或 Railway GPU 实例
```

## 三、风险识别 (Risks)

| # | 风险 | 严重度 | 缓解措施 |
|---|------|--------|---------|
| R1 | **加速度计采样率不足**：普通模式下 wx.onAccelerometerChange 仅 ~5Hz，难以准确检测步频（跑步步频 ~2.5-3.5Hz）。游戏模式虽可达 60Hz，但需申请游戏类目。 | 🔴 高 | 预先测试两种模式的采样率。若 5Hz 满足 Nyquist（> 7Hz 采样足以检测 3.5Hz 步频），则使用普通模式；否则需考虑游戏类目申请或后处理插值算法。 |
| R2 | **AI 推理延迟不可接受**：Railway CPU 实例处理 30 秒视频的 Pose Estimation 可能需要 30-60 秒，用户体验差。 | 🟡 中 | ① 前端用骨架屏 + 预估进度条缓解等待焦虑；② 服务端优化（帧采样、降低分辨率）；③ 如 CPU 推理 >60s 不可接受，Phase 3 升级 GPU。 |
| R3 | **微信小程序审核风险**：如申请游戏类目（为高采样率加速度计），需额外资质；跑步姿态分析可能被归类为"健康"类目，需相关资质。 | 🟡 中 | 当前类目确认 → 检查微信小程序后台"服务类目"。如有不确定，先提交审核试探反馈。跑步训练类目（工具-效率）通常可通过。 |
| R4 | **视频上传大小限制**：wx.uploadFile 限制单文件 ≤10MB。60 秒 1080p 视频远超此限制。 | 🟡 中 | 用 wx.compressVideo 预压缩后再上传；或限制拍摄时长 30 秒 / 分辨率 720p。前端展示文件大小提示。 |
| R5 | **后端 Railway 域名白名单**：小程序正式版必须配置 request 合法域名，Railway 域名需提前录入。 | 🟢 低 | 已确认流程：MP 后台 → 开发 → 开发设置 → 服务器域名。每月可修改 5 次。 |
| R6 | **微信云开发迁移成本**：Phase 2 引入云开发后，用户数据从本地 Storage 迁移到云数据库需设计迁移策略。 | 🟢 低 | 开发阶段用户数据少，迁移简单。使用 wx.getStorageSync → cloud DB 批量同步即可。 |
| R7 | **与 iOS 版功能一致性问题**：iOS 版有端侧 Vision 框架（实时姿态检测），微信版只能做视频上传式分析，功能差距需在产品和营销侧妥善处理。 | 🟡 中 | 在 compare 页和产品文案中明确"微信版提供云端智能分析"，差异化定位为"便捷轻量版"。 |
