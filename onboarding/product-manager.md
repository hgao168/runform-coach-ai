## 三、v1 PRD 概要：RunForm Live Coach

### 产品愿景

> 跑者打开 RunForm → 点击「开始跑步」→ 手机放在腰包/臂带中 → 跑步过程中通过耳机听到实时语音提示："步频 168，偏低，试着加快到 175" / "注意挺直躯干，你有点前倾了"

### Must Have（v1 必须交付，否则不发版）

| # | 功能 | 用户价值 | 技术挑战 |
|---|------|----------|----------|
| M1 | 实时步频检测与语音播报 | 跑者最常见需求：保持理想步频（170-180 spm） | CoreMotion 加速度计数据 → 步态周期识别 → 实时 cadence |
| M2 | 躯干姿态实时监测 | 避免跑步中不自觉前倾/后仰 | 陀螺仪+加速度计融合 → 姿态角估计 |
| M3 | 跑步结束后生成姿态总结报告 | 让用户体验 AI 教练的完整闭环 | 将实时数据聚合为 AnalysisResponse 格式，复用现有分析管线 |
| M4 | 跑步历史记录与趋势对比 | 用户粘性核心：看到进步 | 复用现有 HistoryView，新增「实时跑步」类型 |

### Should Have（v1 尽力交付，可降级到 v1.1）

| # | 功能 | 用户价值 |
|---|------|----------|
| S1 | 步幅/跨步风险评估 | 过度跨步是跑步伤病第一大诱因 |
| S2 | 着地方式提示（前掌/全掌/后跟） | 跑者热门话题，差异化卖点 |
| S3 | 跑步中姿势评分实时更新 | 游戏化体验，激励持续改善 |
| S4 | 自定义目标步频/配速区间 | 满足不同水平跑者需求 |

### Could Have（v2 考虑，v1 不做）

| # | 功能 |
|---|------|
| C1 | Apple Watch 独立 App（腕部姿态数据更准确） |
| C2 | 左右脚对称性实时分析 |
| C3 | 跑步视频 + IMU 数据融合分析 |
| C4 | 伤病风险实时预警（基于步态不对称性） |
| C5 | 实时语音教练对话（用户可用语音提问） |

### 4.2 逐竞品分析

#### Runna（英国，最强训练计划竞品）

- **做什么**：个性化马拉松/跑步训练计划，基于用户能力和目标自动生成
- **强在哪**：计划质量极高，Marathon Block 很专业；品牌认可度在英语跑圈快速上升
- **弱在哪**：完全不做姿态分析；计划是「盲生成」——不知道你的实际跑姿
- **RunForm 差异化**：RunForm 的 v1 训练计划 **基于用户实际姿态数据驱动**——你的髋部下坠 → 臀肌训练 → 计划中自动加入针对性力量训练。Runna 做不到这一点
- **威胁等级**：🟡 中等。赛道不同（计划 vs 姿态），但 Runna 如果未来加入姿态分析会是最大威胁

#### Nike Run Club（NRC，品牌跑步社区）

- **做什么**：免费跑步 App，名人教练语音指导跑，社交挑战，基础数据追踪
- **强在哪**：免费、品牌号召力、教练音频内容质量高、Nike 生态
- **弱在哪**：语音内容是预录制的通用指导，不针对个人跑姿；不做姿态分析
- **RunForm 差异化**：NRC 的"实时语音"是播客式陪伴，RunForm v1 的"实时语音"是 **数据驱动的个性化教练**——告诉你此刻步频多少、是否跨步、躯干是否前倾
- **威胁等级**：🟢 低。功能定位完全不同，NRC 是「陪伴」，RunForm 是「矫正」

#### Strava（全球最大运动社交平台）

- **做什么**：运动追踪、社交分享、路段挑战、训练数据分析
- **强在哪**：社交网络效应极强，数据可视化优秀，API 生态丰富
- **弱在哪**：不做任何姿态分析或实时指导；分析维度限于 GPS/心率/功率等外部指标
- **RunForm 差异化**：Strava 告诉你「跑了多远多快」，RunForm 告诉你「用什么样的身体在跑」——这是根本不同的价值主张
- **威胁等级**：🟢 低。互补而非竞争关系（已有 Strava 同步集成）

#### Keep（中国，最大泛健身平台）

- **做什么**：涵盖跑步/健身/瑜伽/饮食的全品类健身 App
- **强在哪**：中国市场渗透率极高，内容生态完善，社区活跃，微信生态整合
- **弱在哪**：跑步只是其众多品类之一，不做深度跑步姿态分析；跑步功能偏基础
- **RunForm 差异化**：Keep 是「大而全」，RunForm 是「小而精」——只做跑步姿态一件事，做到极致。中国跑者对伤病预防的需求正在快速增长
- **威胁等级**：🟡 中等。如果 Keep 决定深耕跑步姿态赛道，资源碾压是真实风险

#### 咕咚（Codoon，中国最大跑步垂直平台）

- **做什么**：跑步 GPS 追踪、智能训练计划、跑鞋商城、赛事服务
- **强在哪**：跑步垂直用户基数大，品牌在跑圈认知度高，商城变现能力强
- **弱在哪**：也完全不做姿态分析；核心竞争力在 GPS 追踪准确性和跑鞋电商
- **RunForm 差异化**：「咕咚告诉你跑鞋买哪双，RunForm 告诉你穿这双鞋你的跑姿对不对」
- **威胁等级**：🟡 中等。如果咕咚在现有跑鞋推荐中加入姿态分析会是差异化加强

## 五、Top 3+ 风险与缓解

### 风险 1：CoreMotion 姿态精度不足以支撑实时教练 🔴 高

**描述**：手机放在腰包/臂带中时，IMU 传感器远离人体质心，姿态推断精度可能严重下降。Apple Watch 是理想载体但 v1 不考虑。

**缓解方案**：
- Sprint 0 做技术可行性验证：在多种携带方式（腰包/臂带/短裤口袋）下采集数据，与视频 ground truth 对比
- 如果手机精度不够，v1 限制为 "建议手机放腰包（最接近质心）"
- 步频检测只需加速度计峰值检测，精度要求远低于姿态角，优先保证步频可用
- 如果所有携带方式都无法可靠推断躯干角度，v1 范围缩减为「步频语音教练 + 视频分析增强」

### 风险 2：实时语音反馈打断跑步体验 🟡 中

**描述**：过于频繁的语音提示会让跑者烦躁，过于稀疏则失去教练价值。NRC 的成功在于"恰到好处的陪伴"，RunForm 需要在"指导"和"打扰"之间找到平衡。

**缓解方案**：
- Sprint 0 做跑者访谈（至少 5 位不同水平跑者），了解可接受的提示频率和内容偏好
- 设计三级干预模式：「静默模式」（只记录不提示）/「关键提示」（只在姿势明显偏离时提示）/「主动教练」（持续语音反馈）
- v1 默认使用「关键提示」模式，降低骚扰风险
- 提示音设计用温和短促音（类似 Garmin 的警示音），而非长篇大论

### 风险 3：与 Runna 定位混淆，用户获取成本高 🟡 中

**描述**：如果一个跑者在 App Store 搜 "running coach AI"，RunForm 和 Runna 都会出现。Runna 有先发优势和品牌认知，RunForm 需要在用户心智中建立清晰的品类区分。

**缓解方案**：
- 应用描述和截图中强调 "Real-time Form Feedback" 和 "Injury Prevention" 作为核心差异化
- App Store 关键词策略避开直接与 Runna 竞争 "training plan"，专注 "running form" / "pose analysis" / "injury prevention"
- 利用 backlog3.md 中已有的强大定位："AI Running Form & Injury Coach" + "me vs Kipchoge" 作为 hook
- v1 发布时联系跑步垂直媒体/KOL 做评测，强调「第一次有 App 在跑步中告诉你跑姿问题」

### 风险 4：中国市场（微信小程序）与 iOS 主产品不同步 🟡 中

**描述**：微信小程序目前复用部分后端能力，但受限于微信环境无法使用 CoreMotion 等原生传感器，v1 实时教练功能在小程序上可能无法实现。

**缓解方案**：
- v1 实时教练仅在 iOS（和后续 Android）原生 App 上提供
- 微信小程序 v1 定位为「轻量版」：保留视频分析 + 训练计划，新增跑步结束后的数据回顾（从原生 App 同步）
- 在 PRD 中明确标注四端差异

## 六、需要 CEO 拍板的决策

| # | 决策项 | 选项 A | 选项 B | 建议 |
|---|--------|--------|--------|------|
| **D1** | **v1 是否仅限 iOS** | 仅 iOS 发 v1，Android 延后到 v1.1 | iOS + Android 同步开发 v1 | **建议 A**。Android 端目前代码库为空，同步开发会稀释 Sprint 0-1 的 iOS 聚焦度。先用 iOS 验证产品假设，跑通后再移植 |
| **D2** | **实时语音：TTS 合成 vs 预录音频** | 用 AVSpeechSynthesizer 直接在手机端合成中文/英文语音 | 录制一组专业教练的提示音频片段，按场景拼接播放 | **建议 A（v1）+ B（v2）**。TTS 成本零、多语言零成本，v1 快速上线；v2 可根据用户反馈录制品牌化音频 |
| **D3** | **v1 是否加入 Apple Watch 支持** | v1 不依赖 Apple Watch，只用 iPhone IMU 数据 | v1 同步开发腕部 App，获取更准确的手腕姿态数据 | **建议 A**。Watch 开发增加 2-4 周工作量，且限制用户群（需有 Watch）。先用 iPhone 验证核心价值，如果精度不够再在 v1.1 引入 Watch |
| **D4** | **首次定价策略** | 免费试用 7 天 + 订阅制（参考 Runna £15.99/月 但定价更低） | 完全免费 + 高级功能一次性买断 | **建议 A**。订阅制提供可持续收入支持 AI API 成本。定价建议 ¥38/月 或 $4.99/月（Runna 的 1/3），降低中国市场价格敏感度 |
| **D5** | **v1 是否保留 Strava 同步** | 保留现有 Strava 同步功能，v1 不做变更 | 暂时隐藏 Strava 同步，聚焦核心教练体验 | **建议 A**。Strava 同步已开发完成，维护成本低，且是获取海外用户的重要渠道（Strava 用户可以直接导入训练数据后用 RunForm 分析姿态） |
