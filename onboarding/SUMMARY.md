## 二、跨角色共识发现

### 共同确认的核心缺失

1. **无实时运动传感器管线** — iOS（无 CoreMotion）、Android（无 SensorManager）。这是 v1 与现有产品的分水岭。
2. **零测试覆盖** — iOS/Android/后端/微信小程序，共 0 个测试文件。QA 审计：仓库只有 1 个冒烟脚本 `scripts/test_analyzer.py`。
3. **无发布管道** — Android 无签名无 Google Play、微信小程序纯手工上传、Web 端不存在。
4. **实时语音反馈完全缺失** — v1 核心差异化功能为零起点。

### 共同推荐的架构方向

| 方向 | 共识 |
|------|------|
| iOS 传感器 | AsyncSequence / AsyncStream（非 Combine） |
| iOS 最低版本 | iOS 16（覆盖 95%+ 设备） |
| Android UI | 纯 Jetpack Compose（现有代码风格） |
| Android minSdk | 26（保持现状，覆盖 95%） |
| Web 框架 | Next.js 14+（React），视频标注+图表生态获胜 |
| 小程序定位 | 独立完整产品（非 iOS 伴侣），走微信云开发混合架构 |
| 数据持久化 | iOS: Core Data / Android: Room（双方推荐方案） |

## 三、需要 CEO 拍板的三大跨领域决策

### 决策 1：v1 平台交付节奏 —— 资源分配的顶层约束 🔴

**背景**：v1 的「实时跑步语音教练」是五端共享的品类定义功能，但各平台基础不同。

| 选项 | 描述 | 人力需求 | 时间线 |
|------|------|---------|--------|
| **A：iOS 单点突破** | v1 仅 iOS 交付 CoreMotion + 语音教练，Android/微信小程序/Web 延后 | iOS × 1 人全力 | v1 6-8 周 |
| **B：iOS + Android 双端同步** | v1 同时交付 iOS 和 Android 的实时教练 | iOS × 1 + Android × 1 | v1 8-10 周（含同步成本） |
| **C：全平台最小可感知** | iOS 完整版 + Android 基础版（仅步频）+ 微信小程序步频模式 + Web 官网 MVP | iOS + Android + 小程序 + Web 各 1 人 | v1 6-8 周（并行） |

**各角色立场**：
- PM：建议 A（iOS 先验证，Android v1.1）
- iOS 开发：支持 A 或 B，技术上可行
- Android 开发：已有 2,186 行可用代码，双端并行成本不高
- 微信小程序：独立开发流，不与 iOS 抢资源，建议 C 中包含小程序
- QA：建议 B（Android 已有代码，Sprint 1 接入传感器后可内部测试）
- 营销：建议 A（聚焦一个平台做爆点 PR，"跑步中实时语音教练"需要稳定体验）

**推荐**：**A（iOS 单点突破）** + 其他平台 Sprint 1-2 做技术准备（Android 传感器 PoC、微信步频模块、Web 脚手架），但不承诺 v1 发布。理由：v1 的「实时语音教练」是全新品类，体验打磨比平台覆盖更重要。先用 iOS 把算法和 UX 跑通，再移植其他平台可大幅降低多端返工风险。

---

### 决策 2：CoreMotion 实时指标范围 —— Sprint 1 的交付边界 🔴

**背景**：iOS 和 Android 的传感器管线从零开始。PoseExtractor 现有 21 项视频指标，传感器版需要各项独立研发。

**iOS 建议路径**（Phase 0-5，~13-18 天）：

| Phase | 指标 | 天数 |
|-------|------|------|
| 0 | 原始传感器采集（加速度计+陀螺仪，100Hz） | 1-2 |
| 1 | 步频实时计算（低通滤波+峰值检测→SPM） | 2-3 |
| 2 | 垂直振幅 + 触地时间估算 | 2-3 |
| 3 | 躯干倾角检测 + 语音触发逻辑 | 2-3 |
| 4 | 实时语音合成（AVSpeechSynthesizer） | 1-2 |
| 5 | 跑步后总结报告 + 历史集成 | 2-3 |

**选项**：

| 选项 | Sprint 1 交付指标 | 风险 |
|------|-------------------|------|
| **A：MVP 三步走** | Phase 0-3（步频+垂直振幅+触地时间+躯干角） | 低风险，4-7天可验证核心价值 |
| **B：完整对标** | 全部 Phase 0-5（15+ 指标对齐视频版） | 高风险，13-18天且每个指标需独立算法 |

**推荐**：**A（MVP 三步走）**。理由是：(1) 步频是跑者最关注的第一指标（调查显示 80%+ 跑者最关心步频）；(2) 每个新增指标需要 2-3 天算法+验证，全部做完超出 Sprint 1-2 容量；(3) MVP 验证 PMF 后再追加指标，避免「没人在乎触地时间」的过度工程。

---

### 决策 3：质量基础设施投入 —— 技术债务偿还决心 🔴

**背景**：仓库 5 个平台 0 测试覆盖，3 个 CI workflow 全是手动触发且无测试步骤，无 Lint 配置。

**QA 提出的投入清单**：

| 项目 | 成本 | 收益 |
|------|------|------|
| **真机设备采购**（iOS 2台 + Android 3-4台） | ¥15,000-30,000 | 传感器精度验证、步频算法校调——无法用模拟器替代 |
| **CI 测试集成**（GitHub Actions 8 个 workflow） | 约 2-3 天搭建 | 自动化回归、PR 门禁防退化 |
| **Lint 体系**（SwiftLint + ktlint + ruff/eslint） | 约 1 天配置 | 统一代码风格，PR review 减少争议 |
| **测试目标**（Sprint 3 前 60% 单元覆盖率） | 持续投入 | 重构信心、Bug 提前发现 |

**选项**：

| 选项 | 描述 | 风险 |
|------|------|------|
| **A：全面投入** | Sprint 0 搭建 CI+测试+Lint 基础设施，采购设备，设定覆盖率目标 | 短期挤占功能开发时间（约 1 人·周） |
| **B：逐步还债** | Sprint 0 只配 Lint，测试「有机会时」补，设备延后采购 | 测试永远补不上的风险极高，没有强制门禁的测试会快速腐化 |
| **C：最小投入** | 设备用开发机代替，测试靠人工回归，CI 维持现状 | v1 发布前会积累大量 Bug，热修复消耗将超过前期节省的时间 |

**推荐**：**A（全面投入）**。理由：(1) 跑步 App 的传感器 Bug（如步频计算错误导致错误语音提示）对用户体验是毁灭性的——不是「页面排版歪了」级别的 Bug；(2) 当前仓库已从零做到 150+ 文件，再不建立测试体系，后续重构成本指数级增长；(3) Sprint 0 本来就是「搭地基」的 Sprint，没有功能交付压力。
